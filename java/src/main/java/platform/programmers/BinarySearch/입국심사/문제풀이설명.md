# 프로그래머스 - 입국심사 (이분 탐색)

이 문제는 앞서 풀었던 징검다리와 같은 **Parametric Search** 유형입니다. 이번에는 "최솟값을 최대화"가 아니라 **"총 시간을 최소화"**하는 문제이지만, 핵심 사고 구조는 동일합니다.

---

## 1. 왜 이분 탐색인가?

### 완전탐색이 불가능하다

사람이 최대 10억 명, 심사관이 최대 10만 명이므로 시뮬레이션으로 한 명씩 배치하는 방식은 시간 내에 불가능합니다.

### 정답의 범위가 정해져 있다

정답은 "모든 사람이 심사를 마치는 데 걸리는 최소 시간"입니다. 이 값은 **최소 1분, 최대 (가장 느린 심사관의 시간 × n)분** 사이에 반드시 존재합니다.

### 단조성이 존재한다

시간을 mid분이라고 가정했을 때, mid가 **커질수록** 각 심사관이 처리할 수 있는 사람 수가 **늘어납니다.** 따라서 "mid분 안에 n명을 모두 심사할 수 있는가?"라는 질문은 mid가 작으면 불가능, mid가 충분히 크면 가능으로, **경계가 딱 한 번 바뀝니다.**

### mid분 동안 심사 가능한 인원 계산법

핵심 공식은 간단합니다. 심사관의 소요 시간이 t분이면, mid분 동안 이 심사관은 **mid / t명**(소수점 버림)을 심사할 수 있습니다. 예를 들어 mid=28이고 t=7이면 28/7 = 4명, t=10이면 28/10 = 2명입니다. 모든 심사관의 처리 인원을 합산하면 mid분 안에 총 몇 명을 심사할 수 있는지 알 수 있습니다.

---

## 2. 예제 트레이스 (n=6, times=[7,10])

| mid (시간) | 7분 심사관 처리 | 10분 심사관 처리 | 합계 | n=6 가능? |
|---|---|---|---|---|
| 20 | 20/7 = 2명 | 20/10 = 2명 | 4명 | 불가능 |
| 25 | 25/7 = 3명 | 25/10 = 2명 | 5명 | 불가능 |
| 28 | 28/7 = 4명 | 28/10 = 2명 | 6명 | **가능** |
| 30 | 30/7 = 4명 | 30/10 = 3명 | 7명 | 가능 |

불가능, 불가능, ..., **가능**, 가능, ... 경계의 첫 "가능"이 정답입니다. 답은 **28**.

---

## 3. 전체 코드

```java
import java.util.Arrays;

class Solution {
    public long solution(int n, int[] times) {
        // [1단계] 정렬 — 탐색 범위의 상한을 구하기 위함
        Arrays.sort(times);

        // [2단계] 이분 탐색 범위 설정
        // 최소 1분, 최대는 가장 느린 심사관이 혼자 n명을 전부 처리하는 시간
        long left = 1;
        long right = (long) times[times.length - 1] * n;
        long answer = right; // 최악의 경우로 초기화

        // [3단계] 이분 탐색 반복
        while (left <= right) {
            long mid = (left + right) / 2;

            // [4단계] mid분 동안 모든 심사관이 처리할 수 있는 총 인원 계산
            long total = 0;
            for (int time : times) {
                total += mid / time; // 각 심사관이 mid분 안에 처리 가능한 인원

                // 오버플로우 방지: 이미 n명 이상이면 더 계산할 필요 없음
                if (total >= n) break;
            }

            // [5단계] 결과에 따라 탐색 범위 조정
            if (total >= n) {
                // mid분이면 n명 이상 처리 가능 → 정답 후보로 기록, 더 짧은 시간 탐색
                answer = mid;
                right = mid - 1;
            } else {
                // mid분으로는 부족 → 더 긴 시간 탐색
                left = mid + 1;
            }
        }

        return answer;
    }
}
```

## 4. 코드 단계별 상세 해설

### 4-1. 정렬과 범위 설정

```java
Arrays.sort(times);
long right = (long) times[times.length - 1] * n;
```

right는 **최악의 경우**입니다. 가장 느린 심사관(10분) 한 명이 6명을 전부 처리하면 60분이 걸립니다. 정답은 이보다 클 수 없으므로 탐색 상한으로 설정합니다. 여기서 `(long)` 캐스팅이 중요한데, int × int 연산이 int 범위(약 21억)를 초과할 수 있기 때문입니다.

### 4-2. 그리디 판단

```java
long total = 0;
for (int time : times) {
    total += mid / time;
    if (total >= n) break;
}
```

이 부분이 징검다리의 그리디 판단에 해당합니다. 각 심사관이 mid분 동안 처리할 수 있는 인원을 전부 더합니다. `mid / time`은 정수 나눗셈이므로 자동으로 소수점이 버려집니다. 예를 들어 28/10 = 2명(2.8명이 아님)입니다. `total >= n`이면 이미 충분하므로 더 계산할 필요 없이 바로 빠져나옵니다. 이 처리가 없으면 심사관이 10만 명이고 mid가 매우 클 때 total이 long 범위를 넘을 수 있습니다.

### 4-3. 탐색 범위 조정 (징검다리와의 차이)

```java
if (total >= n) {
    answer = mid;
    right = mid - 1;  // 더 작은 시간을 탐색
} else {
    left = mid + 1;   // 더 큰 시간을 탐색
}
```

징검다리에서는 "가능하면 더 큰 값을 시도"(left = mid + 1)했지만, 이 문제에서는 **"가능하면 더 작은 값을 시도"**(right = mid - 1)합니다. 이것은 목표가 다르기 때문입니다. 징검다리는 최솟값을 **최대화**하는 것이었고, 입국심사는 시간을 **최소화**하는 것이기 때문입니다. 방향만 반대일 뿐, "가능한 쪽으로 기록하고 더 좋은 답을 탐색"하는 구조는 동일합니다.

## 5. 징검다리와의 비교

| 구분 | 징검다리 | 입국심사 |
|---|---|---|
| 목표 | 최솟값의 **최대화** | 시간의 **최소화** |
| mid의 의미 | 목표 최소 간격 | 주어진 총 시간 |
| 판단 질문 | "n개 이하 제거로 간격 mid 이상 가능?" | "mid분 안에 n명 심사 가능?" |
| 가능할 때 | answer 기록, **left** = mid+1 | answer 기록, **right** = mid-1 |
| 불가능할 때 | **right** = mid-1 | **left** = mid+1 |
| 시간 복잡도 | O(R log D) | O(T log(max×N)) |

두 문제 모두 Parametric Search의 세 가지 조건(정답 범위 존재, 빠른 판단 가능, 단조성)을 만족하며, 구조가 거의 동일합니다. 핵심 차이는 **최대화냐 최소화냐**에 따라 가능할 때 탐색 방향이 반대라는 점뿐입니다.

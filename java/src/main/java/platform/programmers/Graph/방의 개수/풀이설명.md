## 1. 문제 요약

(0,0)에서 시작하여 8방향(0~7)으로 선을 그을 때, 닫힌 영역(방)의 개수를 구하는 문제이다.

방향은 다음과 같이 정의된다.

```
  7  0  1
   ↖ ↑ ↗
6 ← · → 2
   ↙ ↓ ↘
  5  4  3
```

**제약 조건**: arrows 배열 크기는 1 이상 100,000 이하이며, 원소는 0 이상 7 이하이다.

**예시**: `arrows = [6,6,6,4,4,4,2,2,2,0,0,0,1,6,5,5,3,6,0]` → 결과: `3` (삼각형 1개 + 큰 사각형 1개 + 평행사변형 1개)

## 2. 핵심 알고리즘 분석

이 문제는 **평면 그래프의 사이클(Cycle) 탐지** 문제이다. 오일러의 평면 그래프 공식 `V - E + F = 2`에서 유도할 수 있으며, 실전에서는 간선을 하나씩 추가하며 방의 생성을 점진적으로 탐지하는 접근법을 사용한다.

### 2.1 새로운 방이 생기는 조건

새로운 방이 생기려면 다음 두 가지 조건이 **동시에** 만족해야 한다.

1. **이미 방문한 노드**에 도달한다.
2. 그 노드로 가는 **간선(edge)이 새로운 경로**이다.

왜 두 번째 조건이 필요한가? 만약 A→B→A로 왕복만 한다면, 노드는 재방문하지만 방(닫힌 영역)은 생기지 않는다. 같은 간선을 되돌아간 것뿐이기 때문이다. 기존 노드에 **새로운 경로**로 도달해야만, 기존 경로와 새 경로가 합쳐져서 닫힌 영역을 형성한다.

### 2.2 대각선 교차 문제와 스케일업 해법

단위 1로 이동하면, 대각선끼리 교차할 때 교차점이 노드로 인식되지 않아 방을 놓치게 된다.

```
예시: 모래시계 형태

  A --- B          교차점 X가 노드로 잡히지 않으면
   \  /            실제로 방이 2개인데 0개로 카운트됨
    X  ← 이 점이 누락!
   / \
  C --- D
```

**해결법**: 모든 이동을 **2배로 스케일업**한다. 한 칸 이동할 때 실제로는 2칸(중간 노드 포함)을 이동하면, 대각선이 교차하는 지점이 자연스럽게 중간 노드로 생성되어 교차점이 누락되지 않는다.

### 2.3 단계별 알고리즘 요약

```
1. 방향별 dx, dy 정의 (8방향)
2. 각 arrow에 대해 2번씩 이동 (스케일업)
3. 매 이동마다:
   a. 다음 좌표(nx, ny) 계산
   b. 간선 (현재→다음) 정의
   c. if (다음 노드가 방문됨 AND 간선이 새로운 경로):
        방 개수 +1
   d. 노드, 간선을 방문 처리 (간선은 양방향 모두 저장)
   e. 현재 위치를 다음 위치로 갱신
```

## 3. 솔루션 코드

### 3.1 Java

```java
import java.util.*;

class Solution {
    public int solution(int[] arrows) {
        // 8방향: 0=↑, 1=↗, 2=→, 3=↘, 4=↓, 5=↙, 6=←, 7=↖
        int[] dx = {0, 1, 1, 1, 0, -1, -1, -1};
        int[] dy = {1, 1, 0, -1, -1, -1, 0, 1};

        // 방문한 노드 집합
        Set<String> visitedNodes = new HashSet<>();
        // 방문한 간선 집합 (양방향 모두 저장하여 무방향 간선으로 취급)
        Set<String> visitedEdges = new HashSet<>();

        int rooms = 0;
        int x = 0, y = 0;
        visitedNodes.add(x + "," + y);

        for (int arrow : arrows) {
            // 대각선 교차 문제 해결을 위해 2번 이동 (스케일 x2)
            for (int i = 0; i < 2; i++) {
                int nx = x + dx[arrow];
                int ny = y + dy[arrow];

                String nodeKey = nx + "," + ny;
                // 간선을 "출발점-도착점" 형태로 표현
                String edgeKey = x + "," + y + "-" + nx + "," + ny;
                String edgeKeyReverse = nx + "," + ny + "-" + x + "," + y;

                // 이미 방문한 노드에, 새로운 간선으로 도달 → 방 생성!
                if (visitedNodes.contains(nodeKey) && !visitedEdges.contains(edgeKey)) {
                    rooms++;
                }

                // 노드와 간선 모두 방문 처리
                visitedNodes.add(nodeKey);
                visitedEdges.add(edgeKey);
                visitedEdges.add(edgeKeyReverse); // 양방향 저장 (무방향 간선)

                x = nx;
                y = ny;
            }
        }

        return rooms;
    }
}
```

### 3.3 C++

```cpp
#include <string>
#include <vector>
#include <set>
using namespace std;

int solution(vector<int> arrows) {
    int dx[] = {0, 1, 1, 1, 0, -1, -1, -1};
    int dy[] = {1, 1, 0, -1, -1, -1, 0, 1};

    set<pair<int,int>> visitedNodes;
    set<pair<pair<int,int>, pair<int,int>>> visitedEdges;

    int rooms = 0;
    int x = 0, y = 0;
    visitedNodes.insert({x, y});

    for (int arrow : arrows) {
        for (int i = 0; i < 2; i++) {
            int nx = x + dx[arrow];
            int ny = y + dy[arrow];

            auto node = make_pair(nx, ny);
            auto edge = make_pair(make_pair(x, y), make_pair(nx, ny));
            auto edgeRev = make_pair(make_pair(nx, ny), make_pair(x, y));

            // 이미 방문한 노드 + 새 간선 = 새 방
            if (visitedNodes.count(node) && !visitedEdges.count(edge)) {
                rooms++;
            }

            visitedNodes.insert(node);
            visitedEdges.insert(edge);
            visitedEdges.insert(edgeRev);

            x = nx;
            y = ny;
        }
    }

    return rooms;
}
```

### 3.4 Rust

```rust
use std::collections::HashSet;

fn solution(arrows: Vec<i32>) -> i32 {
    let dx = [0, 1, 1, 1, 0, -1, -1, -1];
    let dy = [1, 1, 0, -1, -1, -1, 0, 1];

    let mut visited_nodes: HashSet<(i64, i64)> = HashSet::new();
    let mut visited_edges: HashSet<(i64, i64, i64, i64)> = HashSet::new();

    let mut rooms = 0;
    let (mut x, mut y): (i64, i64) = (0, 0);
    visited_nodes.insert((x, y));

    for &arrow in &arrows {
        let d = arrow as usize;
        for _ in 0..2 {
            let nx = x + dx[d];
            let ny = y + dy[d];

            let is_visited_node = visited_nodes.contains(&(nx, ny));
            let is_new_edge = !visited_edges.contains(&(x, y, nx, ny));

            if is_visited_node && is_new_edge {
                rooms += 1;
            }

            visited_nodes.insert((nx, ny));
            visited_edges.insert((x, y, nx, ny));
            visited_edges.insert((nx, ny, x, y));

            x = nx;
            y = ny;
        }
    }

    rooms
}
```

## 6. 오일러 평면 그래프 공식 심화

### 6.1 공식의 의미와 직관

**V - E + F = 2**

V는 꼭짓점(Vertex)의 수, E는 간선(Edge)의 수, F는 면(Face)의 수이다. 여기서 "면"이란 간선으로 둘러싸인 영역을 의미하며, 그래프 바깥의 무한한 영역도 하나의 면으로 센다는 것이 중요하다.

가장 단순한 예시부터 확인해 보자.

```
삼각형 하나:

    A
   / \
  B---C

V = 3 (A, B, C)
E = 3 (AB, BC, CA)
F = 2 (삼각형 내부 1개 + 외부 무한 영역 1개)

검증: 3 - 3 + 2 = 2 ✓
```

좀 더 복잡한 경우도 보자.

```
사각형 안에 대각선 하나:

    A --- B
    |  /  |
    | /   |
    C --- D

V = 4 (A, B, C, D)
E = 5 (AB, AC, BC, BD, CD)
F = 3 (삼각형 2개 + 외부 1개)

검증: 4 - 5 + 3 = 2 ✓
```

```
사각형 두 개가 붙어있는 경우:

    A --- B --- E
    |    |    |
    C --- D --- F

V = 6
E = 7
F = 3 (사각형 2개 + 외부 1개)

검증: 6 - 7 + 3 = 2 ✓
```

### 6.2 왜 항상 성립하는가 — 귀납적 증명

그래프를 **하나씩 쌓아가는** 방식으로 증명할 수 있다. 이 증명 과정은 곧 이 문제의 풀이 로직과 직접적으로 연결된다.

**기저 상태**: 점 하나만 있는 경우이다. V=1, E=0, F=1(외부면만 존재)이므로 `1 - 0 + 1 = 2`가 성립한다.

**간선을 하나 추가할 때** 두 가지 경우만 존재한다.

**Case 1 — 새로운 노드로 뻗어가는 간선을 추가:**

```
추가 전:  A       추가 후:  A --- B
V=1, E=0, F=1    V=2, E=1, F=1

ΔV = +1, ΔE = +1, ΔF = 0
→ V-E+F의 변화량: +1 -1 +0 = 0  (불변!)
```

아직 존재하지 않던 노드에 간선을 연결하면, V와 E가 각각 1씩 증가하고 면은 변하지 않는다. 새 면을 만들려면 영역을 "닫아야" 하는데, 새 노드로 뻗어가는 것은 열린 끝이므로 닫히지 않기 때문이다.

**Case 2 — 이미 존재하는 노드에 새 간선을 연결:**

```
추가 전:  A --- B    추가 후:  A --- B
          |                   |  /
          C                   C

V=3, E=2, F=1     V=3, E=3, F=2

ΔV = 0, ΔE = +1, ΔF = +1
→ V-E+F의 변화량: 0 -1 +1 = 0  (불변!)
```

이미 방문한 노드를 새 간선으로 연결하면 기존 면이 둘로 쪼개진다. 기존 영역을 간선이 가로지르면서 두 개의 면으로 분할되는 것이다. 따라서 E와 F가 각각 1씩 증가하고, V는 변하지 않는다.

어느 경우든 `V - E + F`의 값은 변하지 않으므로, 기저 상태의 2를 영원히 유지한다.

### 6.3 이 문제에 공식이 어떻게 적용되는가

우리가 원하는 것은 "방의 개수", 즉 외부 면을 제외한 내부 면의 수이다.

```
V - E + F = 2
F = E - V + 2
내부 면(방의 개수) = F - 1 = E - V + 1    (외부 면 1개 제거)
```

따라서 **방의 개수 = E - V + 1** 이다.

귀납적 증명의 Case 2를 다시 보면, 이미 방문한 노드에 새 간선이 연결될 때 E는 +1 되고 V는 변하지 않으므로, `E - V + 1`의 값이 정확히 1 증가한다. 이것은 곧 "방이 하나 생겼다"는 뜻이다. 이것이 코드에서 다음 조건문이 하는 일이다.

```java
if (visitedNodes.contains(nodeKey) && !visitedEdges.contains(edgeKey)) {
    rooms++;    // Case 2: 기존 노드 + 새 간선 → 새 방!
}
```

### 6.4 점진적 접근법이 직접 계산보다 나은 이유

이론적으로는 그래프를 다 그린 뒤 최종 V와 E를 세서 `E - V + 1`을 계산해도 된다. 하지만 실전에서는 대각선 교차로 인한 **암묵적 꼭짓점과 간선 분할**을 정확히 처리해야 하므로 구현이 매우 복잡해진다.

**직접 계산 방식(Post-hoc)**: 그래프를 전부 그린 뒤, 모든 교차점을 찾고, 간선을 분할하고, 최종 V와 E를 세서 `E - V + 1`을 반환한다. 교차점 탐색에 O(N²)이 필요할 수 있으며, 간선 분할 로직도 까다롭다.

**점진적 방식(Incremental)**: 간선을 하나씩 추가하면서 Case 2가 발생할 때마다 카운터를 올린다. 스케일업(×2)으로 교차점이 자동으로 중간 노드가 되므로 별도의 교차점 처리가 필요 없다. O(N) 시간에 깔끔하게 해결된다.

결론적으로 두 방식은 수학적으로 동일한 결과를 내지만, 점진적 방식이 구현과 효율 모두에서 압도적으로 유리하다.

## 7. 일반화된 오일러 공식: V - E + F = 1 + C

기본 공식 `V - E + F = 2`는 **연결된(connected) 평면 그래프**에서만 성립한다. 그래프가 여러 연결 요소로 나뉘면 일반화된 공식 `V - E + F = 1 + C`를 사용해야 한다. 여기서 C는 연결 요소의 수이다.

### 7.1 연결 요소(Connected Component)란?

그래프 안에서 **서로 이어져 있는 노드들의 덩어리**를 연결 요소라고 한다. 같은 덩어리 안의 노드들은 간선을 따라가면 서로 도달할 수 있고, 다른 덩어리의 노드에는 어떤 경로로도 도달할 수 없다. 현실 세계의 **섬**에 비유할 수 있다. 다리(간선)로 연결된 땅덩어리(노드)끼리는 같은 섬이고, 다리가 없으면 별개의 섬이다.

**연결 요소가 1개인 그래프:**

```
  A --- B --- C
        |
        D --- E

모든 노드가 간선을 따라 서로 도달 가능
→ 연결 요소: 1개 (그래프 전체가 하나의 덩어리)
```

**연결 요소가 2개인 그래프:**

```
  A --- B --- C          F --- G
        |                |
        D --- E          H

  [--- 덩어리 1 ---]    [덩어리 2]
```

A에서 출발하면 B, C, D, E에는 갈 수 있지만, F, G, H에는 어떤 경로로도 갈 수 없다.

**엄밀한 정의**: 그래프 G의 연결 요소란, (1) 집합 안의 모든 노드 쌍이 경로로 연결되고, (2) 더 이상 확장할 수 없는(maximal) 노드의 부분집합이다. 간선이 하나도 없는 고립된 노드도 그 자체로 하나의 연결 요소이다.

**BFS로 연결 요소 개수를 구하는 코드 (Java):**

```java
int countConnectedComponents(int n, List<List<Integer>> adj) {
    boolean[] visited = new boolean[n];
    int components = 0;

    for (int i = 0; i < n; i++) {
        if (!visited[i]) {
            // 새로운 연결 요소 발견! BFS로 이 덩어리를 전부 방문
            components++;
            Queue<Integer> queue = new LinkedList<>();
            queue.add(i);
            visited[i] = true;

            while (!queue.isEmpty()) {
                int cur = queue.poll();
                for (int next : adj.get(cur)) {
                    if (!visited[next]) {
                        visited[next] = true;
                        queue.add(next);
                    }
                }
            }
        }
    }
    return components;
}
```

### 7.2 연결 요소가 늘어나면 공식이 바뀌는 이유

핵심은 **외부 면(무한 영역)의 공유**에 있다.

```
분리된 삼각형 + 사각형을 같은 평면에 배치:

    A           D---E
   / \          |   |
  B---C         F---G
  
      ← 외부 면은 1개로 합쳐짐! →

V = 3 + 4 = 7
E = 3 + 4 = 7
F = 1(삼각형 내부) + 1(사각형 내부) + 1(공유된 외부) = 3

검증: 7 - 7 + 3 = 3 = 1 + C (C=2) ✓
```

각 연결 요소가 독립적으로 존재하면 외부 면을 1개씩 가지지만, 같은 평면으로 합치면 외부 면이 하나로 합쳐진다. 연결 요소를 합칠 때마다 외부 면 1개가 중복 제거되는 것이다.

**수식으로 유도:**

```
각 연결 요소에 독립적으로 오일러 공식을 적용하여 전부 더하면:
  V_total - E_total + F_각자의_합 = 2C

F_각자의_합에는 외부 면이 C개 포함되어 있으나,
같은 평면에서 외부 면은 실제로 1개뿐이므로:
  F_실제 = F_각자의_합 - (C - 1)

따라서:
  V - E + F_실제 = 2C - (C - 1) = C + 1
```

### 7.3 귀납적 재검증 — 세 가지 Case

노드만 있고 간선이 없는 상태(`V개 노드, 0개 간선, C=V, F=1`)에서 시작하여 간선을 하나씩 추가할 때 세 가지 경우가 발생한다.

**Case A — 서로 다른 연결 요소를 잇는 간선 (두 섬을 다리로 연결):**

```
ΔE = +1, ΔC = -1, ΔF = 0
V-E+F 변화: -1    |    1+C 변화: -1    → 등식 유지 ✓
```

**Case B — 같은 연결 요소 내에서 새 노드로 뻗는 간선 (나뭇가지 확장):**

```
ΔV = +1, ΔE = +1, ΔC = 0, ΔF = 0
V-E+F 변화: 0    |    1+C 변화: 0    → 등식 유지 ✓
```

**Case C — 같은 연결 요소 내에서 기존 노드로 새 간선 (사이클 형성 → 방 생성!):**

```
ΔV = 0, ΔE = +1, ΔC = 0, ΔF = +1
V-E+F 변화: 0    |    1+C 변화: 0    → 등식 유지 ✓
```

세 가지 경우 모두에서 `V - E + F = 1 + C`가 보존된다.

### 7.4 이 문제에서 왜 일반화 공식이 필요 없는가

"방의 개수" 문제에서는 (0,0)에서 출발하여 선을 **끊지 않고** 연속으로 그린다. 따라서 그려지는 그래프는 항상 연결 그래프이고 C=1이 보장된다.

```
방의 개수 = F - 1 = (E - V + 1 + C) - 1 = E - V + C = E - V + 1
```

만약 문제가 "여러 개의 출발점에서 독립적으로 그림을 그릴 때 방의 총 개수"와 같은 변형이었다면 C > 1이 될 수 있으므로 일반화 공식을 적용해야 한다.

## 8. Q&A: 왜 edgeKeyReverse도 Set에 저장하는가?

### 8.1 문제의 본질 — 무방향 간선

우리가 그리는 것은 **선분**이다. A에서 B로 가든, B에서 A로 가든 같은 선분이다. 즉 "A→B"와 "B→A"는 물리적으로 동일한 간선이다. 그러나 코드에서 간선을 문자열로 표현하면 `"0,0-1,1"`과 `"1,1-0,0"`은 서로 다른 문자열이 된다. 양방향을 모두 저장함으로써 "이 간선은 방향이 없는 선분이다"라는 사실을 코드로 표현하는 것이다.

### 8.2 저장하지 않으면 생기는 버그

```
단계 1: (0,0)에서 (1,1)로 이동
  edgeKey = "0,0-1,1"  → Set에 추가
  Set 상태: { "0,0-1,1" }

단계 2: (1,1)에서 (0,0)으로 되돌아감
  edgeKey = "1,1-0,0"  → Set에 있나? 없다!
  nodeKey = "0,0"       → 방문한 노드인가? 맞다!
  
  조건: 방문한 노드 ✓ AND 새로운 간선 ✓
  → rooms++ 🚨 잘못된 카운트!
```

A에서 B로 갔다가 그대로 B에서 A로 되돌아온 것뿐인데, 방이 하나 생겼다고 오판한다. 이것은 같은 선 위를 왕복한 것이지, 영역을 감싸서 닫은 것이 아니다.

### 8.3 저장했을 때 올바르게 동작하는 이유

**왕복 시 (방이 생기면 안 되는 경우):**

```
단계 1: (0,0)에서 (1,1)로 이동
  "0,0-1,1" + "1,1-0,0" 모두 Set에 추가

단계 2: (1,1)에서 (0,0)으로 되돌아감
  edgeKey = "1,1-0,0" → Set에 있다! (이미 저장됨)
  
  조건: 방문한 노드 ✓ AND 새로운 간선 ✗
  → rooms 증가 안 함 ✓ 정확!
```

**삼각형을 그릴 때 (방이 생겨야 하는 경우):**

```
단계 1: (0,0) → (1,0) → 간선 "0,0-1,0" + "1,0-0,0" 저장. (1,0) 새 노드 → 변화 없음
단계 2: (1,0) → (1,1) → 간선 "1,0-1,1" + "1,1-1,0" 저장. (1,1) 새 노드 → 변화 없음
단계 3: (1,1) → (0,0) → "1,1-0,0"이 Set에 있나? 없다! (새로운 간선)
                          (0,0)은 방문한 노드? 맞다!
  → rooms++ ✓ 삼각형이 닫혔으므로 방 1개 생성. 정확!
```

# 프로그래머스 Level 4 — 사칙연산 (문제 1843) 복습 문서

---

## 1. 문제 요약

숫자와 `+`, `-` 연산자로 이루어진 수식에서, **괄호를 어떤 순서로 묶느냐**에 따라 결과가 달라진다. 모든 가능한 괄호 배치 중 **최댓값**을 구해야 한다.

**제약 조건:**

- 숫자 개수: 2 ~ 101개 (연산자는 숫자 - 1개)
- 숫자 범위: 1 ~ 1,000 (자연수, 문자열로 주어짐)
- 배열 길이: 3 ~ 201 (숫자와 연산자가 번갈아 배치)

**입출력 예시:** `["1", "-", "3", "+", "5", "-", "8"]` → 답: `1`

가능한 괄호 배치 중 `(1 - (3 + (5 - 8))) = 1` 또는 `(1 - ((3 + 5) - 8)) = 1`이 최대다.

## 2. 시간 복잡도 상한 추론 — 문제를 읽자마자 알고리즘 급을 판단하는 방법

코딩 테스트에서 문제를 받으면, 제약 조건의 N 크기만 보고 "어떤 복잡도 급의 알고리즘을 써야 하는가"를 빠르게 판단할 수 있다. 이 추론은 두 가지 지식에 기반한다.

### 2-1. 경험칙: 1초에 대략 10⁸ 연산

대부분의 온라인 저지(프로그래머스, LeetCode 등)에서 제한 시간은 1~2초이고, 현대 컴퓨터는 1초에 대략 10⁸(1억) 번의 단순 연산을 처리할 수 있다는 경험칙이 있다. 이 기준에서 역으로 판단하면 이렇다.

- O(10⁶) → 여유롭게 통과
- O(10⁷) → 보통 통과
- O(10⁸) → 빡빡하지만 가능
- O(10⁹) 이상 → 거의 확실히 시간 초과

### 2-2. N의 크기에서 허용 복잡도를 역추론

이 문제에서 숫자 개수 N ≤ 101이다. 다양한 복잡도에 대입하면 이렇게 된다.

- O(N) = 101 → 당연히 OK
- O(N²) = 약 10,201 → 매우 여유
- **O(N³) = 약 1,030,301 ≈ 10⁶** → 10⁸ 한도의 1%도 안 쓰므로 **충분히 여유**
- O(N⁴) = 약 10⁸ → 빡빡하지만 가능할 수도
- O(2ᴺ) = 2¹⁰¹ ≈ 10³⁰ → 완전히 불가능

따라서 "N ≤ 101이면 O(N³)까지는 안전하다"는 결론이 나온다. 구간 DP가 정확히 O(N³)이므로 채택 가능.

### 2-3. 완전탐색이 왜 폭발하는가

괄호를 어디에 칠지 결정하는 문제는, N개의 피연산자를 이진 트리로 묶는 경우의 수를 세는 것과 같다. 이 경우의 수는 카탈란 수(Catalan Number)로 알려져 있으며, 점근적으로 `C(n) ≈ 4ⁿ / (n^(3/2) × √π)` 정도로 증가한다. N=20이면 약 17억으로 이미 시간 초과, N=101이면 10⁵⁶ 수준으로 천문학적이다.

### 2-4. 실전 나침반 — N의 범위가 출제자의 의도를 암시한다

| N의 범위 | 출제자가 기대하는 복잡도 급 |
|---|---|
| N ≤ 10~20 | O(2ᴺ), O(N!), 백트래킹 |
| N ≤ 100~500 | O(N³) |
| N ≤ 5,000~10,000 | O(N²) |
| N ≤ 10⁵~10⁶ | O(N log N) 또는 O(N) |
| N ≤ 10⁷ 이상 | O(N) 또는 O(1) 수학적 풀이 |

이 표는 절대적이지 않지만, "어떤 카테고리의 알고리즘을 떠올려야 하는가"를 빠르게 좁혀주는 나침반 역할을 한다. 이 사칙연산 문제에서 N ≤ 101이 주어진 순간, "O(N³)짜리 알고리즘을 찾아야겠구나"라는 방향이 바로 잡힌다.

## 3. 두 개의 인덱스 공간 — 혼동하기 쉬운 핵심 포인트

이 문제에서는 **숫자 배열**과 **연산자 배열**이 별도로 존재하고, 각각의 인덱스 체계가 다르다. 원래 입력 `["1", "-", "3", "+", "5", "-", "8"]`을 파싱하면 이렇게 분리된다.

```
숫자 인덱스:    num[0]=1    num[1]=3    num[2]=5    num[3]=8
연산자 인덱스:        op[0]=-     op[1]=+     op[2]=-
```

수식을 풀어쓰면 `num[0] op[0] num[1] op[1] num[2] op[2] num[3]`이다. 연산자 `op[k]`는 항상 `num[k]`와 `num[k+1]` **사이에** 위치한다. 연산자의 인덱스 범위는 0 ~ n-2, 숫자의 인덱스 범위는 0 ~ n-1이다.

### DP 테이블과 분할점의 인덱스 공간

`maxDP[i][j]`에서 `i`와 `j`는 항상 **숫자의 인덱스**를 가리킨다. 분할점 `k`는 **연산자의 인덱스**를 가리킨다. 같은 변수명 `k`가 두 인덱스 공간을 넘나드는데, 실제로는 "연산자 `op[k]`는 숫자 `num[k]`와 `num[k+1]` 사이에 있다"는 일대일 대응 덕분에 숫자적으로 문제가 없다.

### 분할의 정확한 의미

"연산자 `op[k]`를 마지막에 적용한다"는 것은, 전체 수식을 `op[k]`를 기준으로 왼쪽 덩어리와 오른쪽 덩어리로 쪼갠 뒤, 각 덩어리를 먼저 계산하고, 마지막에 `op[k]`로 두 결과를 합친다는 뜻이다.

```
전체:  num[0] op[0] num[1] op[1] num[2] op[2] num[3]

op[1](='+')을 마지막에 적용하면:
  왼쪽 덩어리:  num[0] op[0] num[1]              ← 숫자 인덱스 [0..1]
  연결 연산자:  op[1] = '+'
  오른쪽 덩어리: num[2] op[2] num[3]              ← 숫자 인덱스 [2..3]

  → 괄호 표현: (1 - 3) + (5 - 8)
```

패턴을 정리하면, 연산자 `op[k]`를 기준으로 분할할 때 **왼쪽 덩어리는 숫자 `num[0]`~`num[k]`**, **오른쪽 덩어리는 숫자 `num[k+1]`~`num[n-1]`**을 포함한다. `op[k]` 자체는 어느 쪽에도 속하지 않고 두 덩어리를 **연결하는 역할**만 한다.

## 4. 풀이 전략 — 구간 DP

### 4-1. 왜 구간 DP인가

수식 `a op₀ b op₁ c op₂ d`에서 "어디에 괄호를 칠 것인가"는 곧 "어느 연산자를 마지막에 적용할 것인가"와 같다. 마지막으로 적용할 연산자 `op[k]`를 기준으로 수식은 왼쪽 부분식(숫자 `[i..k]`)과 오른쪽 부분식(숫자 `[k+1..j]`)으로 분할된다. 각 부분식도 재귀적으로 같은 구조를 가지므로, **중복 부분 문제 + 최적 부분 구조**를 만족하여 DP 적용이 가능하다.

### 4-2. 왜 최솟값도 추적해야 하는가

핵심은 **뺄셈 연산자**에 있다. 뺄셈에서 결과를 최대화하려면 `최대(A - B) = max(A) - min(B)`이다. 즉, 빼는 쪽(오른쪽)이 **최소**여야 결과가 최대가 된다. 마찬가지로 뺄셈 결과를 최소화하려면 `최소(A - B) = min(A) - max(B)`이다.

따라서 모든 구간에 대해 **maxDP(최댓값)**과 **minDP(최솟값)**을 동시에 관리해야 한다. maxDP만 추적하면 뺄셈 구간에서 잘못된 답이 나온다.

### 4-3. 점화식

숫자 배열 `num[0..n-1]`, 연산자 배열 `op[0..n-2]`에 대해 다음과 같이 정의한다.

**기저 조건:**

```
maxDP[i][i] = minDP[i][i] = num[i]    (숫자 하나짜리 구간)
```

**전이 (i < j일 때, 분할점 k를 i부터 j-1까지 순회):**

```
op[k]가 '+' 인 경우:
    maxDP[i][j] = max(maxDP[i][j],  maxDP[i][k] + maxDP[k+1][j])
    minDP[i][j] = min(minDP[i][j],  minDP[i][k] + minDP[k+1][j])

op[k]가 '-' 인 경우:
    maxDP[i][j] = max(maxDP[i][j],  maxDP[i][k] - minDP[k+1][j])
    minDP[i][j] = min(minDP[i][j],  minDP[i][k] - maxDP[k+1][j])
```

**답:** `maxDP[0][n-1]`

### 4-4. 점화식이 성립하는 직관

덧셈에서는 양쪽 모두 최대화해야 합이 최대가 된다 → `max + max`. 자명하다.

뺄셈에서는 "왼쪽은 크게, 오른쪽은 작게"해야 차이가 최대가 된다 → `max - min`. 예를 들어 `A - B`에서 A가 가능한 한 크고 B가 가능한 한 작으면 결과가 최대다. 이것이 `maxDP[i][k] - minDP[k+1][j]`의 의미다.

## 5. 3중 루프의 구조 — len, i, j 각각의 역할과 초기화/설정 이유

이 절이 이 문제에서 가장 이해하기 어려웠던 부분이다. 천천히 풀어서 설명한다.

### 5-1. DP 테이블이 무엇을 저장하는가

`maxDP[i][j]`는 "숫자 `num[i]`부터 `num[j]`까지의 부분 수식에서, 괄호를 최적으로 배치했을 때 얻을 수 있는 최댓값"이다. 이 테이블의 모든 칸을 올바르게 채우면, `maxDP[0][n-1]`이 전체 수식의 최댓값, 즉 최종 답이 된다.

### 5-2. 이 테이블을 어떤 순서로 채워야 하는가 — 건물 비유

10층짜리 건물을 짓는다고 생각해보자. 10층을 지으려면 9층이 이미 있어야 하고, 9층을 지으려면 8층이 있어야 한다. 1층부터 순서대로 올라가야 한다. DP 테이블도 정확히 같은 원리다.

`maxDP[0][3]`(전체 수식)을 계산하려면, 그 안에 포함된 더 짧은 구간들의 답이 이미 준비되어 있어야 한다. `maxDP[0][3]`의 계산 과정에서 `maxDP[0][0]`, `maxDP[1][3]`, `maxDP[0][1]`, `maxDP[2][3]` 같은 값들을 참조하기 때문이다. 이 값들이 아직 `-∞`(초기값)인 상태에서 `maxDP[0][3]`을 계산하면 엉터리 답이 나온다.

"더 짧은 구간"이란, 구간에 포함된 숫자 개수가 더 적은 것이다. `[0,3]`은 숫자 4개, `[1,3]`은 3개, `[0,1]`은 2개, `[2,2]`는 1개. 이 "숫자 개수"가 곧 **`len`**이다.

따라서 건물을 1층부터 짓듯이, `len`이 작은 구간부터 채워야 한다. `len=1`(1층) → `len=2`(2층) → `len=3`(3층) → `len=4`(4층, 최종 답).

### 5-3. `len`은 왜 2부터 시작하는가

`len=1`인 경우는 숫자가 하나뿐인 구간이다. 연산자가 없으니까 점화식을 적용할 필요가 없고, 그냥 `maxDP[i][i] = nums[i]`로 값을 대입하면 된다. 이 처리는 루프 이전에 별도로 수행한다.

```java
// len=1 처리 (루프 이전)
for (int i = 0; i < n; i++) {
    maxDP[i][i] = nums[i];
    minDP[i][i] = nums[i];
}
```

이미 `len=1`이 완료되었으므로, 3중 루프는 `len=2`부터 시작한다. `len=2`부터가 "연산자가 최소 1개 존재해서 실제로 점화식을 적용해야 하는 구간"의 시작이다.

참고로, `len=1`을 이 루프 안에 넣으면 `j = i + 1 - 1 = i`가 되고, 셋째 루프 `for (k = i; k < j; ...)` 에서 `k < j`가 `k < i`가 되어 루프 본문이 한 번도 실행되지 않는다. 에러는 나지 않지만 아무 일도 하지 않는 빈 반복이 된다.

### 5-4. `len`은 왜 `n`까지 가는가

`len=n`이 되면 구간 `[0, n-1]`이 되어, 배열의 모든 숫자를 포함하는 전체 수식을 의미한다. 이것이 최종적으로 답을 구하고 싶은 구간이므로, `len`은 `n`까지 올라가야 한다.

### 5-5. `i`는 왜 0부터 `n - len`까지인가

`i`는 구간의 시작점이다. 숫자 배열의 인덱스가 0부터 시작하므로 `i`도 0부터 시작한다.

상한이 `n - len`인 이유는 `j`가 배열 밖으로 나가면 안 되기 때문이다. `j = i + len - 1`이고, `j ≤ n - 1`이어야 하므로 이걸 `i`에 대해 풀면 다음과 같다.

```
j ≤ n - 1
i + len - 1 ≤ n - 1
i ≤ n - len
```

구체적으로 `n=4`일 때 각 `len`에서 `i`가 움직이는 범위를 보면 이렇다.

```
len=2: i는 0, 1, 2 (n-len = 4-2 = 2)
  i=0 → j=1  구간 [0,1]  수식 "1 - 3"
  i=1 → j=2  구간 [1,2]  수식 "3 + 5"
  i=2 → j=3  구간 [2,3]  수식 "5 - 8"

len=3: i는 0, 1 (n-len = 4-3 = 1)
  i=0 → j=2  구간 [0,2]  수식 "1 - 3 + 5"
  i=1 → j=3  구간 [1,3]  수식 "3 + 5 - 8"

len=4: i는 0 (n-len = 4-4 = 0)
  i=0 → j=3  구간 [0,3]  수식 "1 - 3 + 5 - 8" (전체)
```

`len`이 커질수록 `i`가 움직일 수 있는 범위가 좁아진다. 구간이 길어질수록 "시작점을 뒤로 밀 여유"가 줄어드는 것이다.

### 5-6. `j = i + len - 1`은 어디서 나온 공식인가

`j`는 독립적인 루프 변수가 아니라, `i`와 `len`으로부터 계산되는 종속 변수다. `len`의 정의에서 직접 나온다.

`len`은 "구간에 포함된 숫자의 개수"이고, 구간 `[i, j]`에 포함된 숫자의 개수는 `j - i + 1`개다. 예를 들어 `[2, 4]`라면 인덱스 2, 3, 4 총 3개이므로 `4 - 2 + 1 = 3`이다. 이걸 `len`으로 놓으면 `len = j - i + 1`이고, `j`에 대해 풀면 `j = i + len - 1`이다.

따라서 바깥 루프에서 `len`을 고정하고, 안쪽 루프에서 `i`를 움직이면, `j`는 그냥 따라오는 종속 변수다. **이 코드를 "`i`와 `j` 두 개의 독립 변수"로 읽으면 헷갈리고, "`len`과 `i` 두 개의 독립 변수, `j`는 종속 변수"로 읽으면 구조가 명확해진다.**

### 5-7. 왜 `i`와 `j`를 독립적으로 2중 루프 돌리지 않는가

`for i = 0..n-1`, `for j = i..n-1` 순서로 돌면, `[0,0]` → `[0,1]` → `[0,2]` → `[0,3]` → `[1,1]` → `[1,2]` → ... 이런 순서가 된다. 여기서 `[0,3]`을 계산할 때 `[1,3]`이 아직 계산되지 않은 상태다. `[1,3]`은 `i=1, j=3`인데, `i`가 1인 행은 아직 방문하지 않았기 때문이다. 이러면 `maxDP[1][3]`이 초기값(`-∞`)인 채로 참조되어 틀린 답이 나온다.

반면 `len`을 바깥 루프로 두면, `len=1`(대각선) → `len=2`(대각선 한 칸 위) → `len=3` → `len=4` 순서로 채워진다. 큰 구간을 계산할 때 참조하는 작은 구간들은 항상 이전 `len` 단계에서 이미 채워져 있으므로, 순서가 자동으로 보장된다.

### 5-8. 이 순서가 안전한 이유 — 한 문장 정리

`maxDP[i][j]`를 계산할 때 참조하는 `maxDP[i][k]`와 `maxDP[k+1][j]`는 원래 구간보다 **항상 짧은 구간**이고, 짧은 구간 = 더 작은 `len` = 이전 단계에서 이미 채워진 칸이기 때문에, `len`을 작은 것부터 키우는 순서로 채우면 **참조 시점에 값이 항상 준비되어 있음이 보장**된다.

### 5-9. DP 테이블의 채우기 순서 시각화

테이블에서 `maxDP[i][j]`의 위치를 보면, `len = j - i + 1`이 같은 칸들은 같은 대각선 위에 놓인다.

```
     j=0    j=1    j=2    j=3
i=0 [len=1] [len=2] [len=3] [len=4]   ← 이 방향으로 채워진다
i=1         [len=1] [len=2] [len=3]      (대각선 → 오른쪽 위 모서리)
i=2                 [len=1] [len=2]
i=3                         [len=1]
```

주대각선(len=1)에서 시작해서 오른쪽 위 모서리(len=n)까지 대각선 한 줄씩 채우는 것이다.

## 6. 트레이스: 예시 입력의 전체 실행 과정

입력: `["1", "-", "3", "+", "5", "-", "8"]`

`nums = [1, 3, 5, 8]`, `ops = ['-', '+', '-']`, `n = 4`

### len=1 (기저 조건)

연산 없이 값 대입만 수행한다.

```
maxDP[0][0] = minDP[0][0] = 1
maxDP[1][1] = minDP[1][1] = 3
maxDP[2][2] = minDP[2][2] = 5
maxDP[3][3] = minDP[3][3] = 8
```

### len=2

**구간 [0,1], 수식 "1 - 3":** 분할점 k=0, op[0]='-'

```
maxDP[0][1] = maxDP[0][0] - minDP[1][1] = 1 - 3 = -2
                ↑ len=1에서 채움    ↑ len=1에서 채움
minDP[0][1] = minDP[0][0] - maxDP[1][1] = 1 - 3 = -2
```

**구간 [1,2], 수식 "3 + 5":** 분할점 k=1, op[1]='+'

```
maxDP[1][2] = maxDP[1][1] + maxDP[2][2] = 3 + 5 = 8
minDP[1][2] = minDP[1][1] + minDP[2][2] = 3 + 5 = 8
```

**구간 [2,3], 수식 "5 - 8":** 분할점 k=2, op[2]='-'

```
maxDP[2][3] = maxDP[2][2] - minDP[3][3] = 5 - 8 = -3
minDP[2][3] = minDP[2][2] - maxDP[3][3] = 5 - 8 = -3
```

### len=3

**구간 [0,2], 수식 "1 - 3 + 5":** 분할점 k=0과 k=1 두 가지 선택지.

```
k=0 (op '-'): max = maxDP[0][0] - minDP[1][2] = 1 - 8 = -7
              min = minDP[0][0] - maxDP[1][2] = 1 - 8 = -7
                      ↑ len=1        ↑ len=2

k=1 (op '+'): max = maxDP[0][1] + maxDP[2][2] = -2 + 5 = 3
              min = minDP[0][1] + minDP[2][2] = -2 + 5 = 3
                      ↑ len=2         ↑ len=1

→ maxDP[0][2] = max(-7, 3) = 3
→ minDP[0][2] = min(-7, 3) = -7
```

**구간 [1,3], 수식 "3 + 5 - 8":** 분할점 k=1과 k=2.

```
k=1 (op '+'): max = maxDP[1][1] + maxDP[2][3] = 3 + (-3) = 0
k=2 (op '-'): max = maxDP[1][2] - minDP[3][3] = 8 - 8 = 0

→ maxDP[1][3] = 0, minDP[1][3] = 0
```

### len=4 (최종 답)

**구간 [0,3], 전체 수식 "1 - 3 + 5 - 8":** 분할점 k=0, k=1, k=2.

```
k=0 (op '-'): max = maxDP[0][0] - minDP[1][3] = 1 - 0 = 1      ← len=1, len=3 참조
k=1 (op '+'): max = maxDP[0][1] + maxDP[2][3] = -2 + (-3) = -5  ← len=2, len=2 참조
k=2 (op '-'): max = maxDP[0][2] - minDP[3][3] = 3 - 8 = -5     ← len=3, len=1 참조

→ maxDP[0][3] = max(1, -5, -5) = 1  ← 최종 답
```

### maxDP 최종 테이블

```
     j=0  j=1  j=2  j=3
i=0 [ 1 ] [-2 ] [ 3 ] [ 1 ]  ← maxDP[0][3] = 1 (답)
i=1       [ 3 ] [ 8 ] [ 0 ]
i=2             [ 5 ] [-3 ]
i=3                   [ 8 ]
```

### minDP 최종 테이블

```
     j=0  j=1  j=2  j=3
i=0 [ 1 ] [-2 ] [-7 ] [-15]
i=1       [ 3 ] [ 8 ] [ 0 ]
i=2             [ 5 ] [-3 ]
i=3                   [ 8 ]
```

### JavaScript

```javascript
function solution(arr) {
    // 숫자와 연산자를 분리
    const nums = [];
    const ops = [];
    for (let i = 0; i < arr.length; i++) {
        if (i % 2 === 0) nums.push(parseInt(arr[i]));
        else ops.push(arr[i]);
    }

    const n = nums.length;
    const INF = 1e15;

    // maxDP[i][j]: 숫자 인덱스 i~j 구간에서 얻을 수 있는 최댓값
    // minDP[i][j]: 숫자 인덱스 i~j 구간에서 얻을 수 있는 최솟값
    const maxDP = Array.from({ length: n }, () => Array(n).fill(-INF));
    const minDP = Array.from({ length: n }, () => Array(n).fill(INF));

    // len=1: 기저 조건 — 숫자 하나짜리 구간은 자기 자신이 최대이자 최소
    for (let i = 0; i < n; i++) {
        maxDP[i][i] = nums[i];
        minDP[i][i] = nums[i];
    }

    // len=2부터 n까지: 작은 구간부터 채워야 큰 구간 계산 시 참조값이 준비됨
    for (let len = 2; len <= n; len++) {
        // i: 구간 시작점. i + len - 1 ≤ n - 1 이어야 하므로 i ≤ n - len
        for (let i = 0; i <= n - len; i++) {
            // j: 구간 끝점. len과 i로부터 자동 결정되는 종속 변수
            const j = i + len - 1;
            // k: 분할점(연산자 인덱스). 구간 [i,j]를 [i,k]와 [k+1,j]로 분할
            for (let k = i; k < j; k++) {
                if (ops[k] === "+") {
                    // 덧셈: 양쪽 최대끼리 더하면 최대, 최소끼리 더하면 최소
                    maxDP[i][j] = Math.max(maxDP[i][j], maxDP[i][k] + maxDP[k + 1][j]);
                    minDP[i][j] = Math.min(minDP[i][j], minDP[i][k] + minDP[k + 1][j]);
                } else {
                    // 뺄셈: 최대 - 최소 = 최대, 최소 - 최대 = 최소
                    maxDP[i][j] = Math.max(maxDP[i][j], maxDP[i][k] - minDP[k + 1][j]);
                    minDP[i][j] = Math.min(minDP[i][j], minDP[i][k] - maxDP[k + 1][j]);
                }
            }
        }
    }

    // 전체 구간 [0, n-1]의 최댓값이 답
    return maxDP[0][n - 1];
}
```

func solution(arr []string) int {
    // 숫자와 연산자를 분리

            minDP[i][j] = inf

### 코드 변형: 기저 조건을 루프 안에 통합하기

위 코드들은 모두 `len=1`(기저 조건)을 별도의 초기화 루프로 처리하고, 3중 루프는 `len=2`부터 시작한다. 기저 조건을 3중 루프 안에 통합할 수도 있다. `len=1`일 때 `if` 분기로 값을 대입하고 `continue`로 `k` 루프를 건너뛰면 된다.

```java
// 별도 초기화 루프를 제거하고, 아래 루프 안에 통합한 버전

for (int len = 1; len <= n; len++) {       // ← 2가 아니라 1부터 시작
    for (int i = 0; i <= n - len; i++) {
        int j = i + len - 1;

        // 기저 조건: 숫자 하나짜리 구간은 자기 자신이 최대이자 최소
        if (len == 1) {
            maxDP[i][j] = nums[i];
            minDP[i][j] = nums[i];
            continue;  // k 루프에 진입할 필요 없음
        }

        for (int k = i; k < j; k++) {
            if (ops[k] == '+') {
                maxDP[i][j] = Math.max(maxDP[i][j], maxDP[i][k] + maxDP[k + 1][j]);
                minDP[i][j] = Math.min(minDP[i][j], minDP[i][k] + minDP[k + 1][j]);
            } else {
                maxDP[i][j] = Math.max(maxDP[i][j], maxDP[i][k] - minDP[k + 1][j]);
                minDP[i][j] = Math.min(minDP[i][j], minDP[i][k] - maxDP[k + 1][j]);
            }
        }
    }
}
```

`continue`를 넣은 이유: `len=1`이면 `j = i`이므로 `k` 루프의 조건 `k < j`가 `k < i`가 되어 어차피 한 번도 실행되지 않는다. `continue`가 없어도 동작은 동일하지만, "기저 조건은 여기서 끝"이라는 의도를 명시적으로 보여준다.

두 방식은 결과가 완전히 동일하다. 기저 조건을 별도로 빼는 쪽이 "len=1은 특별하다"는 점을 구조적으로 드러내므로 가독성 면에서 더 일반적이지만, 루프 안에 통합하는 쪽이 코드가 더 짧아지는 장점이 있다.

## 8. 엣지 케이스

| 관점 | 케이스 | 솔루션 처리 |
|---|---|---|
| 최솟값 경계 | 숫자 2개 (N=2), 예: `["5", "-", "3"]` | 구간 길이 2에서 바로 처리. 분할점 1개만 존재하므로 정상 동작 |
| 최댓값 경계 | 숫자 101개, 모두 1000 | DP 테이블 101×101, O(101³) ≈ 10⁶ 연산으로 충분. 값 범위: 최대 ±101,000 → long 사용으로 안전 |
| 모든 연산자가 + | `["1", "+", "2", "+", "3"]` | 덧셈은 결합법칙이 성립하므로 어떤 순서든 동일. maxDP = minDP = 6 |
| 모든 연산자가 - | `["10", "-", "3", "-", "2"]` | 순서에 따라 결과가 달라짐. `(10-(3-2))=9` vs `((10-3)-2)=5`. maxDP가 9를 올바르게 반환 |

## 9. 복잡도 분석

| 풀이 | 시간 복잡도 | 공간 복잡도 | 비고 |
|---|---|---|---|
| 구간 DP (최적 해법) | O(N³) | O(N²) | N = 숫자 개수 (최대 101). 약 10⁶ 연산으로 충분히 여유 |
| 완전 탐색 (DFS) | O(카탈란 수) ≈ O(4ⁿ) | O(N) | N=101일 때 천문학적. 시간 초과 확실 |

### 9-1. 구간 DP가 정확히 O(N³)인 이유 — 3중 루프 반복 횟수 직접 세기

"3중 루프니까 O(N³)"이라고 대략 감을 잡을 수도 있지만, 실제로 안쪽 루프 본문이 총 몇 번 실행되는지를 정확히 세어보면 O(N³)이 나오는 과정을 확인할 수 있다.

안쪽 루프의 점화식은 비교, 덧셈/뺄셈, 대입뿐이므로 한 번 실행에 O(1)이다. 따라서 총 시간 복잡도는 안쪽 루프의 본문이 총 몇 번 실행되느냐에 달려 있다.

각 `len`에서, 중간 루프의 `i`는 `(n - len + 1)`번 반복하고, 각 `i`에 대해 안쪽 루프의 `k`는 `j - i = len - 1`번 반복한다. 따라서 특정 `len`에서의 안쪽 루프 실행 횟수는 `(n - len + 1) × (len - 1)`이다.

`n=4`인 경우에 구체적으로 세어보면 이렇다.

```
len=2: i는 3번 반복(0,1,2), 각각 k가 1번 → 3 × 1 = 3번
len=3: i는 2번 반복(0,1),   각각 k가 2번 → 2 × 2 = 4번
len=4: i는 1번 반복(0),     각각 k가 3번 → 1 × 3 = 3번
                                           합계: 10번
```

일반적으로 전체 합을 수식으로 쓰면 이렇다.

```
총 실행 횟수 = Σ(len=2 ~ n) (n - len + 1) × (len - 1)
```

`len - 1`을 `m`으로 치환하면(m은 1부터 n-1까지) 이렇게 변한다.

```
= Σ(m=1 ~ n-1) (n - m) × m
= Σ(m=1 ~ n-1) (nm - m²)
= n × Σm - Σm²
= n × n(n-1)/2 - n(n-1)(2n-1)/6
```

최고차항만 보면 `n³/2 - n³/3 = n³/6`이 된다. 상수 계수 1/6은 빅오 표기에서 무시하므로, 총 시간 복잡도는 **O(N³)**이다.

직관적으로는, 3중 루프 각각의 범위가 최대 N이므로 상한은 N × N × N = N³이다. 실제로는 `len`이 커지면 `i`의 범위가 줄어들고, `len`이 작으면 `k`의 범위가 줄어들기 때문에 정확한 횟수는 N³/6 정도로 N³보다 작다. 하지만 빅오 표기에서는 상수 계수를 무시하므로 O(N³)이라고 쓴다.

`n=101`을 대입하면 실제 연산 횟수는 대략 `101³/6 ≈ 171,700`번 정도로, 10⁸ 한도에 비해 매우 여유롭다.

## 10. 핵심 포인트 요약 — 시험 직전 30초 복기용

1. **이 문제의 본질:** "어느 연산자를 마지막에 적용할 것인가"를 선택하는 것이 곧 괄호 배치다.
2. **왜 구간 DP:** 분할 지점 k를 기준으로 왼쪽 [i..k]와 오른쪽 [k+1..j]가 독립적으로 재귀 → 중복 부분 문제 + 최적 부분 구조 → DP.
3. **왜 minDP도 필요:** 뺄셈에서 `최대 = max - min`. 오른쪽의 최솟값을 알아야 뺄셈 결과를 최대화할 수 있다.
4. **채우기 순서:** `len`(구간 길이)을 1 → 2 → ... → n으로 키우며, 대각선 방향으로 테이블을 채운다. 큰 구간은 작은 구간의 답에 의존하므로 이 순서가 필수다. (`i`, `j`를 독립 2중 루프로 돌리면 아직 안 채운 칸을 참조하여 틀린 답이 나온다.)
5. **두 개의 인덱스 공간:** `i`, `j`는 숫자 인덱스, `k`는 연산자 인덱스. `op[k]`는 `num[k]`와 `num[k+1]` 사이에 위치한다. `j`는 독립 변수가 아니라 `j = i + len - 1`로 결정되는 종속 변수다.
6. **시간 복잡도 판단:** N ≤ 101 → O(N³) ≈ 10⁶ → 1초 제한(~10⁸)의 1%이므로 충분히 여유. (정확한 실행 횟수는 N³/6이지만, 빅오에서 상수 계수 무시.)
7. **`len`이 2부터 시작하는 이유:** `len=1`은 연산자가 없어 점화식 적용이 불필요하다. 루프 이전에 `maxDP[i][i] = nums[i]`로 별도 처리하거나, 루프 안에서 `if (len == 1) { 대입; continue; }`로 통합할 수 있다.
8. **`i`의 상한이 `n - len`인 이유:** `j = i + len - 1 ≤ n - 1`이어야 하므로, `i ≤ n - len`이 된다. `len`이 커질수록 `i`의 범위가 좁아진다.

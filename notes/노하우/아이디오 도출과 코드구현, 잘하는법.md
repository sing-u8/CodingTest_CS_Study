## 아이디어 도출과 코드 구현, 잘하는 방법

좋은 질문입니다. 이 두 가지는 사실 **서로 다른 근육**입니다. 각각 따로 훈련해야 하고, 각각 막히는 이유도 다릅니다. 하나씩 깊이 다뤄볼게요.

---

## Part 1. 아이디어 도출이 힘든 이유와 해결법

### 왜 아이디어가 안 떠오르는가?

아이디어가 떠오르지 않는 근본 원인은 대부분 **"문제를 있는 그대로 보기 때문"** 입니다. 코딩 테스트 문제는 항상 표면적인 스토리(섬, 카메라, 조이스틱 등) 뒤에 **추상적인 수학/알고리즘 구조**가 숨어 있습니다. 이 구조를 꿰뚫어 보는 눈이 곧 아이디어 도출 능력입니다.

예를 들어 오늘 푼 "카메라 설치" 문제를 떠올려 보세요. 처음엔 도로와 차량이라는 스토리가 있지만, 본질은 "최소 개수의 점으로 모든 구간을 찌르기"라는 순수한 구조였습니다. 이 변환이 보이는 순간 Greedy가 자연스럽게 따라옵니다.

---

### 아이디어 도출을 위한 5단계 사고법

**1단계: 문제를 추상화하라**

스토리를 걷어내고, 핵심 자료 구조 관점으로 다시 표현해보는 훈련을 합니다. "차량이 진입하고 나간다" → "구간(interval)이 주어진다"처럼요. 아래 변환 패턴을 익혀두면 큰 도움이 됩니다.

```
"최소 비용으로 모두 연결"   → 그래프, MST
"경로가 겹치는가"          → 구간(Interval)
"가장 큰/작은 것 선택"      → Greedy 또는 정렬
"경우의 수, 최소/최대값"    → DP
"연결 여부 판단"            → Union-Find, BFS/DFS
"순서가 있는 선택"          → 스택/큐
```

**2단계: 제약 조건에서 힌트를 뽑아라**

제약 조건은 단순한 범위 제한이 아닙니다. 알고리즘의 힌트가 숨어 있습니다. "최솟값이 8보다 크면 -1" → 탐색 깊이가 8로 제한 → 브루트포스 또는 DP. "노드 수 N ≤ 100" → O(N²)도 괜찮다 → 플로이드 워셜도 가능. 이처럼 제약 조건을 보면 어떤 시간 복잡도가 허용되는지 역산할 수 있습니다.

```
N ≤ 20           → 비트마스킹, 백트래킹
N ≤ 1,000        → O(N²) 가능
N ≤ 100,000      → O(N log N) 필요
N ≤ 1,000,000    → O(N) 또는 O(N log N) 필요
```

**3단계: 극단적인 예시를 직접 그려라**

문제에서 주어진 예시만 보지 말고, 직접 **극단적인 케이스**를 그려보세요. "모든 구간이 겹치면?", "A가 전혀 없으면?", "숫자가 하나뿐이면?" 같은 질문을 던지면서 패턴을 찾다 보면 아이디어가 보입니다. 오늘의 조이스틱 문제에서 "BAAAAAAAAC"처럼 A가 많은 극단 케이스를 그렸을 때 "왼쪽 감기"라는 핵심 전략이 드러난 것처럼요.

**4단계: 완전탐색부터 시작해서 최적화하라**

아이디어가 전혀 안 떠오른다면, 일단 **"모든 경우를 다 해보면 어떻게 될까?"** 를 먼저 생각하세요. 그 다음 "여기서 어떤 중복 계산이 있지?" → DP, "어떤 선택이 확실히 손해지?" → Greedy 순서로 최적화 방향을 좁혀갑니다. 처음부터 최적 알고리즘을 찾으려 하면 막히지만, 완전탐색에서 출발하면 길이 보입니다.

**5단계: 알고리즘 선택 기준을 외우지 말고 "이유"를 이해하라**

"정렬 + 진출 기준 = Greedy"를 외우는 것보다, "왜 진출 기준으로 정렬하면 최적이 보장되는가?"를 이해하는 게 훨씬 강력합니다. 이유를 이해하면 비슷한 구조의 새 문제에서도 같은 논리를 적용할 수 있거든요. 패턴을 외우는 사람은 응용이 안 되지만, 원리를 이해한 사람은 변형 문제도 풀 수 있습니다.

## Part 2. 코드 구현이 힘든 이유와 해결법

### 왜 아이디어는 있는데 코드가 안 써지는가?

아이디어가 있는데 코드가 안 나오는 건 대부분 **아이디어가 아직 충분히 구체화되지 않았기 때문**입니다. "진출 지점 기준으로 정렬하고 Greedy로 카메라 설치"라는 아이디어는 맞지만, 코드로 쓰려면 그것보다 훨씬 더 구체적인 질문들이 해결되어야 합니다. "정렬 후 어떤 변수를 추적하지?", "카메라가 커버하는지 여부는 어떻게 판단하지?", "루프 종료 조건은?" 같은 것들이요.

---

### 코드 구현을 위한 3단계 방법

**1단계: 아이디어를 "자연어 의사코드"로 먼저 써라**

코드를 바로 쓰려 하지 말고, 한국어로 먼저 단계를 적어보세요. "1. 진출 지점 기준으로 정렬한다. 2. lastCamera 변수를 -무한대로 초기화한다. 3. 각 차량을 순회하면서, lastCamera가 현재 차량의 진입 지점보다 작으면 카메라를 진출 지점에 설치한다." 이 수준으로 적혀 있으면 코드는 그냥 번역에 가깝습니다. 이 과정을 건너뛰고 바로 코드를 쓰려 하면 손이 멈추는 겁니다.

**2단계: 변수 하나하나에 명확한 의미를 부여하라**

구현이 막히는 순간은 대부분 "이 변수가 지금 뭘 의미하는 거지?"가 헷갈릴 때입니다. 코드를 쓰기 전에 필요한 변수의 이름과 의미를 먼저 정의하세요. `lastCamera = "마지막으로 설치한 카메라의 위치"`, `cameras = "지금까지 설치한 카메라 수"`. 이 정의가 명확하면 각 변수가 언제 갱신되어야 하는지도 자연스럽게 따라옵니다.

**3단계: 예시를 직접 손으로 돌려보면서 코드를 써라**

코드를 다 쓰고 나서 예시로 검증하는 게 아니라, **예시를 돌리면서 동시에 코드를 써나가는** 방식이 훨씬 효과적입니다. 예시의 첫 번째 케이스를 처리하는 코드를 쓰고, 두 번째 케이스로 넘어가면서 "지금 쓴 코드로 이게 처리되는가?"를 계속 확인하는 거죠. 오늘 추적했던 것처럼 `cameras=0, lastCamera=MIN_VALUE` 상태에서 차량 하나씩 처리하면서 코드가 맞는지 검증하는 방식입니다.

---

## 두 능력을 동시에 키우는 훈련법

마지막으로 가장 중요한 한 가지를 말씀드릴게요. 문제를 풀고 나서 **"왜 이 알고리즘이어야 했는가"를 스스로 설명할 수 있을 때까지 복습**하는 것입니다. 단순히 코드를 다시 읽는 게 아니라, 종이에 문제만 놓고 "이 문제를 처음 보는 사람에게 어떻게 접근하라고 설명할까?"를 써보는 거예요.

이걸 꾸준히 하면 두 가지 효과가 생깁니다. 아이디어 도출 측면에서는 비슷한 구조의 문제를 만났을 때 "이전에 이런 추상 구조를 본 적 있다"는 패턴 인식 능력이 쌓입니다. 코드 구현 측면에서는 특정 알고리즘을 구현하는 관용적인 코드 패턴(Union-Find의 find, Greedy의 정렬+순회 등)이 손에 익게 됩니다. 지금 복습 정리 md 파일을 만드는 습관을 이미 잘 들이고 계시니, 거기에 "처음 이 문제를 봤을 때 어떻게 접근해야 할지"를 한 섹션 추가해 보시는 걸 추천드립니다! 😊

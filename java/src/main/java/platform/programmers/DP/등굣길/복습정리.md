# 프로그래머스 42898 — 등굣길 (Level 3, DP)

> **분류**: 동적 계획법 (Dynamic Programming)  
> **핵심 알고리즘**: 격자 경로 수 세기 (Grid Path Counting)  
> **시간 복잡도**: O(m × n) | **공간 복잡도**: O(m × n)  
> **문제 링크**: https://school.programmers.co.kr/learn/courses/30/lessons/42898

---

## 1. 문제 요약

m×n 크기의 격자에서 집 `(1,1)`에서 학교 `(m,n)`까지 **오른쪽과 아래쪽으로만** 이동하여 도달할 수 있는 **최단경로의 개수**를 구하는 문제이다. 단, 물에 잠긴 지역(puddles)은 지나갈 수 없으며, 결과를 `1,000,000,007`로 나눈 나머지를 반환해야 한다.

**제약 조건**:
- m, n은 1 이상 100 이하의 자연수
- puddles의 원소는 `[열, 행]` 순서 ← **주의!**
- puddles의 길이는 0 이상 10 이하

---

## 2. 핵심 통찰: 왜 DP인가?

### 2-1. 모든 유효 경로 = 최단경로

"오른쪽과 아래쪽으로만 이동"이라는 조건이 핵심이다. 이 조건 덕분에 어떤 경로를 선택하든 반드시 오른쪽으로 `(m-1)`번, 아래쪽으로 `(n-1)`번 이동하게 된다. 되돌아가는 경우가 없으므로 **모든 경로의 길이가 동일**하고, 따라서 "최단경로의 개수"는 곧 "가능한 모든 경로의 개수"와 같다.

### 2-2. 최적 부분 구조 (Optimal Substructure)

어떤 셀 `(i, j)`에 도달하려면 반드시 **왼쪽 셀 `(i, j-1)`** 또는 **위쪽 셀 `(i-1, j)`**에서 와야 한다. 따라서 `(i, j)`까지의 경로 수는 이 두 셀까지의 경로 수를 더한 것과 같다. 이것이 DP 적용의 근거가 되는 최적 부분 구조이다.

### 2-3. 점화식

```
dp[i][j] = dp[i-1][j] + dp[i][j-1]    (웅덩이가 아닌 경우)
dp[i][j] = 0                            (웅덩이인 경우)
```

**기저 조건**: `dp[1][1] = 1` (출발점에 도달하는 방법은 1가지 — 제자리)

### 2-4. 웅덩이 처리

웅덩이인 셀을 `dp[i][j] = 0`으로 고정하면, 이후 해당 셀을 경유하는 경로가 자연스럽게 0으로 전파된다. 별도의 복잡한 분기 처리가 필요 없는 것이 DP 풀이의 장점이다.

---

## 3. 예시 트레이스 (m=4, n=3, puddles=[[2,2]])

### 3-1. 좌표 변환

puddles `[2, 2]`는 열=2, 행=2를 의미하므로 → `dp[2][2]`가 웅덩이이다.

### 3-2. DP 테이블 채우기 과정

**Step 1 — 초기화 & 웅덩이 마킹**:

```
     c1   c2   c3   c4
r1 [  1 ] [  0 ] [  0 ] [  0 ]
r2 [  0 ] [  X ] [  0 ] [  0 ]
r3 [  0 ] [  0 ] [  0 ] [  0 ]
```

`dp[1][1] = 1`, `dp[2][2] = -1`(웅덩이 마킹)

**Step 2 — 첫 행과 첫 열 채우기** (점화식 적용 결과):

```
     c1   c2   c3   c4
r1 [  1 ] [  1 ] [  1 ] [  1 ]    ← 첫 행: 왼쪽에서만 올 수 있음
r2 [  1 ] [  X ] [  0 ] [  0 ]
r3 [  1 ] [  0 ] [  0 ] [  0 ]
        ↑ 첫 열: 위쪽에서만 올 수 있음
```

**Step 3 — 나머지 셀 채우기**:

```
     c1   c2   c3   c4
r1 [  1 ] [  1 ] [  1 ] [  1 ]
r2 [  1 ] [  0 ] [  1 ] [  2 ]
r3 [  1 ] [  1 ] [  2 ] [  4 ]  ← 정답!
```

**계산 과정 상세**:

| 셀 | 계산식 | 결과 |
|---|---|---|
| `dp[2][2]` | 웅덩이 → 0 | **0** |
| `dp[2][3]` | `dp[1][3] + dp[2][2]` = 1 + 0 | **1** |
| `dp[2][4]` | `dp[1][4] + dp[2][3]` = 1 + 1 | **2** |
| `dp[3][2]` | `dp[2][2] + dp[3][1]` = 0 + 1 | **1** |
| `dp[3][3]` | `dp[2][3] + dp[3][2]` = 1 + 1 | **2** |
| `dp[3][4]` | `dp[2][4] + dp[3][3]` = 2 + 2 | **4** |

웅덩이 `dp[2][2] = 0`이 어떻게 전파되는지에 주목하자. `dp[2][3]`을 계산할 때 왼쪽 값이 0이 되어, 웅덩이를 경유하는 경로가 자연스럽게 제거된다.

## 4. 솔루션 코드

### Java

```java
class Solution {
    public int solution(int m, int n, int[][] puddles) {
        final int MOD = 1_000_000_007;
        
        // dp[r][c]: (1,1)에서 (r,c)까지의 경로 수
        // 인덱스 1부터 사용하여 경계 처리를 단순화
        int[][] dp = new int[n + 1][m + 1];
        
        // 웅덩이를 -1로 마킹
        // ★ puddles[i]는 [열, 행] 순서이므로 dp[행][열]로 변환
        for (int[] p : puddles) {
            dp[p[1]][p[0]] = -1;
        }
        
        // 출발점 초기화
        dp[1][1] = 1;
        
        // 좌→우, 상→하로 테이블 채우기
        for (int r = 1; r <= n; r++) {
            for (int c = 1; c <= m; c++) {
                if (r == 1 && c == 1) continue;
                
                // 웅덩이는 0으로 (통과 불가)
                if (dp[r][c] == -1) {
                    dp[r][c] = 0;
                    continue;
                }
                
                // 위쪽 + 왼쪽
                long up   = (r > 1) ? dp[r - 1][c] : 0;
                long left  = (c > 1) ? dp[r][c - 1] : 0;
                dp[r][c] = (int)((up + left) % MOD);
            }
        }
        
        return dp[n][m];
    }
}
```

### Go

```go
func solution(m, n int, puddles [][]int) int {
    const MOD = 1_000_000_007

    dp := make([][]int, n+1)
    for i := range dp {
        dp[i] = make([]int, m+1)
    }

    // ★ puddles는 [열, 행] 순서
    for _, p := range puddles {
        dp[p[1]][p[0]] = -1
    }

    dp[1][1] = 1

    for r := 1; r <= n; r++ {
        for c := 1; c <= m; c++ {
            if r == 1 && c == 1 {
                continue
            }
            if dp[r][c] == -1 {
                dp[r][c] = 0
                continue
            }
            up, left := 0, 0
            if r > 1 {
                up = dp[r-1][c]
            }
            if c > 1 {
                left = dp[r][c-1]
            }
            dp[r][c] = (up + left) % MOD
        }
    }

    return dp[n][m]
}
```

fn solution(m: i32, n: i32, puddles: Vec<Vec<i32>>) -> i32 {

## 5. 코드 로직 단계별 해설

**1단계 — dp 테이블 생성**: `(n+1) × (m+1)` 크기의 2차원 배열을 0으로 초기화한다. 인덱스를 1부터 사용하면 `r-1`이나 `c-1`이 0일 때 자동으로 0이 되므로, 별도의 경계 체크 없이 점화식을 적용할 수 있다.

**2단계 — 웅덩이 마킹**: puddles 배열을 순회하면서 해당 위치를 `-1`로 표시한다. `puddles[i] = [열, 행]`이므로, `dp[행][열]`에 마킹해야 한다.

**3단계 — 출발점 초기화**: `dp[1][1] = 1`로 설정한다.

**4단계 — 테이블 채우기**: 이중 for문으로 왼쪽 위에서 오른쪽 아래로 순회하며 점화식을 적용한다. 이 순회 순서가 보장하는 것은, `dp[r][c]`를 계산할 때 `dp[r-1][c]`와 `dp[r][c-1]`이 이미 확정되어 있다는 것이다.

**5단계 — 결과 반환**: `dp[n][m]`이 정답이다.

## 6. 함정과 실수 포인트

### ⚠️ 좌표 반전 (가장 흔한 실수)

문제에서 `m`은 가로(열 수), `n`은 세로(행 수)이고, puddles도 `[열, 행]` 순서로 주어진다. 일반적인 배열 인덱싱 `arr[행][열]`과 **정반대**이므로, puddle `[a, b]`는 반드시 `dp[b][a]`에 매핑해야 한다.

```
문제의 좌표:  (col, row) = (2, 2)
배열의 인덱스: dp[row][col] = dp[2][2]    ← 이 경우 우연히 같지만, 개념적으로 반전 필요
```

### ⚠️ MOD 연산 누락

중간 계산에서 MOD를 빠뜨리면 int 범위를 초과할 수 있다. 합산 시 `long` 타입을 사용하거나 매 단계 MOD를 취해야 한다. `(a + b) % p = ((a % p) + (b % p)) % p`이므로 중간에 MOD를 취해도 결과에 영향 없다.

### ⚠️ 첫 행/첫 열에서 웅덩이 뒤의 셀들

첫 행에서 중간에 웅덩이가 있으면 그 오른쪽 셀들은 모두 0이 된다 (왼쪽에서만 올 수 있는데, 왼쪽이 0이므로). 첫 열도 마찬가지이다. 이는 점화식이 자연스럽게 처리해주므로 별도 코딩이 필요 없지만, 디버깅 시 이 현상을 이해하고 있어야 혼란이 없다.

## 7. 복잡도 분석

| 항목 | 분석 |
|---|---|
| **시간 복잡도** | O(m × n) — 격자의 모든 셀을 정확히 한 번씩 방문 |
| **공간 복잡도** | O(m × n) — dp 테이블 크기 |
| **공간 최적화** | 한 행씩 처리하면 O(m)으로 축소 가능 (1차원 DP rolling array) |

---

## 8. 수학적 관점 — 웅덩이 없는 경우

웅덩이가 없다면 이 문제는 순수한 조합론 문제이다. 오른쪽 `(m-1)`번, 아래쪽 `(n-1)`번의 이동을 나열하는 것이므로, 답은 이항계수 `C(m+n-2, m-1)`이다. 하지만 웅덩이가 있으면 특정 경로가 차단되어 조합론적 풀이가 복잡해지고(포함-배제 원리 등), DP가 훨씬 자연스럽고 범용적인 접근법이 된다.

## 9. 자가 점검 Q&A

**Q1. 왜 BFS/DFS가 아니라 DP인가?**

BFS/DFS로 모든 경로를 탐색하면 시간 복잡도가 지수적으로 폭발한다 (최악의 경우 C(200, 100)가지 경로). DP는 중복 부분 문제를 재활용하여 O(m×n)으로 해결한다. 이 문제는 "경로의 개수"를 세는 것이지 "경로를 찾는 것"이 아니므로, 탐색보다 계수(counting)에 특화된 DP가 적합하다.

**Q2. 점화식에서 왜 `dp[i-1][j] + dp[i][j-1]`인가?**

셀 `(i, j)`에 도착하는 마지막 한 걸음은 반드시 "위에서 내려오기" 또는 "왼쪽에서 오른쪽으로 가기" 둘 중 하나이다. 이 두 경우는 서로 배타적(mutually exclusive)이므로, 합의 법칙(Addition Principle)에 의해 두 값을 더하면 된다.

**Q3. MOD를 매 단계 취해도 괜찮은 이유는?**

모듈러 산술의 성질 중 하나인 `(a + b) mod p = ((a mod p) + (b mod p)) mod p`가 보장하기 때문이다. 덧셈 연산에 대해 모듈러 연산은 분배 가능하므로, 중간 결과에 MOD를 취해도 최종 결과는 동일하다.

**Q4. 순회 순서가 왜 중요한가?**

`dp[r][c]`를 계산하려면 `dp[r-1][c]`(위쪽)와 `dp[r][c-1]`(왼쪽)이 이미 확정되어 있어야 한다. 행을 위에서 아래로, 열을 왼쪽에서 오른쪽으로 순회하면 이 의존성이 자연스럽게 충족된다. 만약 순서를 바꾸면(예: 오른쪽→왼쪽) 아직 계산되지 않은 값을 참조하게 되어 오답이 나온다.

**Q5. 1차원 DP로 공간을 최적화할 수 있는가?**

가능하다. 한 행씩 처리할 때, 현재 행의 `dp[c]`를 갱신하면 `dp[c]` 자체가 "이전 행의 같은 열 값"(위쪽)을 이미 담고 있고, `dp[c-1]`은 "현재 행의 이전 열 값"(왼쪽)이 된다. 따라서 1차원 배열 하나로 충분하다. 단, 가독성과 디버깅 용이성을 위해 2차원을 쓰는 것이 일반적이다.
